1.      成为开发者

开发者即第三方服务的提供者，只要是平台的合法用户，都可以是开发者，按照正常的用户注册流程完成注册即可。

2.      创建一个服务

任何一个第三方服务对平台的来说，就是一个 url 地址，其可以是个 html 页面，也可以是一个web api，同时这个 url 地址也是该服务的入口。在服务被授权后，其可通过平台的 web api 来实现与平台的交互，例如获取用户的设备、监测数据等等。

规则与约定：

a)     第三方服务的url地址（服务的入口），必须支持 http get。

如果该服务的运行模式是“后台自动调用(针对api类型服务)”，则还需支持 http post，在此种模式下，平台将发起 http post 请求来访问该服务，并且如果服务是按次数收费，则也是以 http post 的次数为准，不对 http get 进行控制。


b)     参数说明

serviceKey：当前服务的标识。由第三方服务自行存储。第三方服务添加至平台时会生成此标识，是个固定不变的值。

tokenId：标识服务的调用者。平台会将该参数添加至请求中。

callId: 标识了一个按次计费请求。当该第三方服务是按调用次数计费时，平台会将该参数添加至请求中。

 

第三方服务调用平台的web api时，serviceKey及tokeId是必须带上的参数。

3.      创建服务帮助文档

每个第三方服务，都必须有详尽的帮助文档，以便用户在使用前，能了解到该服务的功能、界面等信息。服务要想通过审核，帮助文档必须满足如下条件：

a)     帮助文档对应的 url 可访问。

b)     帮助文档内容详尽，并与实际一致（界面、功能）。

4.      添加服务至平台

服务创建好后，便可添加至平台，在“我的服务”中，有具体的流程引导用户如何添加。

 

在添加过程中，有部分信息（标*）的填入务必保证准确，审核通过后是无法再修改的。特别是第四步，开发者需要正确选择该服务所需要的web api权限，在服务中使用未授权的 api 将会导致调用失败。

添加完成后即会自动提交至系统审核。此时，可以查看到此服务的 serviceKey。

 

5.      测试服务

服务通过审核后，即可在“服务商店”中看到该服务，对于开发者自己所添加的服务，可直接使用。此时，即可用自己的数据对该服务进行测试了，访问其在线效果。

6.      服务认证

服务通过审核后，在必要的情况下，可以对服务进行认证，服务认证的好处如下：

a)     平台级的信誉保证。

b)     增加开发者的信誉度，从而提高用户对此服务的认可。

平台在认证过程中，会对服务的可用性、功能与说明的一致性等方面对服务进行详尽的测试。目前服务认证是收费的（认证失败并按要求修改了服务，再次提交的认证申请是不收取费用的），且认证前需要在“我的账号”完成用户的实名认证：
